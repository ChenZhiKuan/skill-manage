# Review Checklist

这份 checklist 用于在本地检查脚本之后，做一轮严格的人工或 agent review。

## Review Contract

- 输入必须包含：
  - 改动目标
  - 涉及文件
  - diff
  - 预期行为变化
- 输出必须采用 findings-first 风格
- 每条 finding 必须说明：
  - 严重级别
  - 文件位置
  - 风险点
  - 为什么它重要
- 如果没有问题，明确写 `No findings`

## Severity

- `P1`：功能错误、数据损坏、服务无法启动、主要路径不可用
- `P2`：明显回归风险、结构性错误、边界处理缺失、接口不一致
- `P3`：可维护性下降、命名或职责混乱、轻微体验问题

## Scope

本项目当前 review 重点：

- `app.py`
  - 路由是否清晰
  - 扫描路径是否合理
  - Markdown 渲染是否引入明显错误
  - API 输出是否稳定
- `static/app.js`
  - 搜索与弹窗行为是否直接、可预测
  - 事件绑定是否会产生空引用或异常
- `static/style.css`
  - 布局变更是否破坏主要浏览路径
  - 视觉调整是否引入明显可用性问题
- `README.md`
  - 启动路径、命令、端口、能力说明是否仍然准确

## Required Review Questions

### 1. 功能正确性

- 改动是否真的满足目标，而不是只“看起来完成”
- 首页、分类切换、详情页、简介弹窗是否仍可用
- `/api/health` 与 `/api/skills` 是否保持正确

### 2. 回归风险

- 是否破坏了路径扫描逻辑
- 是否引入新的启动依赖
- 是否改变了接口结构但没有同步文档

### 3. 结构一致性

- 是否出现无必要抽象
- 是否出现本可以复用却重复实现的逻辑
- 是否把简单页面工具改成难维护结构

### 4. 前后端边界

- 后端返回的数据结构是否与前端消费保持一致
- 新增字段是否真的被使用
- 前端是否依赖未声明的后端细节

### 5. 运行安全与本地边界

- 是否扩大了本地扫描范围但没有说明
- 是否引入危险 shell 调用
- 是否把用户输入直接拼进危险上下文

## Execution Order

1. 先跑 `scripts/review_harness.sh`
2. 再看 diff 和改动目标
3. 再按本 checklist 做 findings-first review
4. 有 `P1` 或 `P2` 时阻断，不进入下一步
5. 修复后重新跑脚本，再做一轮复审
